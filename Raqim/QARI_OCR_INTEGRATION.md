# دمج Qari OCR داخل مشروع رقيم

تم تعديل محرك التعرف البصري في `Backend/ocr_model.py` ليستخدم نموذج **Qari OCR** من عائلة نماذج `NAMAA-Space/Qari-OCR` قدر الإمكان، مع الحفاظ على واجهة التطبيق الحالية للرفع والمراجعة وحفظ التصحيحات وتنزيل النص المصحح.

## طريقة التشغيل الحالية

المحرك الجديد يحافظ على الدالة نفسها التي يستخدمها الباكند:

```python
ocr_with_highlighting(file_path, output_folder)
```

ويعيد بنية الصفحات والكلمات نفسها التي تتوقعها صفحة المراجعة. لأن Qari OCR يعيد نص الصفحة ولا يعيد إحداثيات كل كلمة، يقوم المحرك بتوليد صناديق كلمات تقريبية من النص المستخرج حتى لا ينكسر مسار المراجعة اليدوية في الواجهة.

| الإعداد | القيمة الافتراضية | الغرض |
|---|---:|---|
| `QARI_OCR_ENABLED` | `true` | تفعيل محاولة استخدام Qari OCR أولًا. |
| `QARI_OCR_PROVIDER` | `space` | استخدام واجهة Hugging Face Space بدل التحميل المحلي الثقيل. |
| `QARI_SPACE_ID` | `oddadmix/Arabic-OCR-Models-Demos` | Space عام يحتوي خيار `Qari OCR 0.2.2.1`. |
| `QARI_SPACE_MODEL_CHOICE` | `Qari OCR 0.2.2.1` | اسم خيار Qari في واجهة Space. |
| `QARI_OCR_FALLBACK_TO_TESSERACT` | `true` | العودة إلى Tesseract إذا فشل Qari بسبب الحصة أو الشبكة أو الموارد. |
| `HF_TOKEN` | غير مضبوط | اختياري؛ يرفع حدود Hugging Face ZeroGPU عند استخدام Space. |

## سبب عدم الاعتماد على التشغيل المحلي داخل الديمو

تمت تجربة تشغيل Qari محليًا عبر Transformers، لكن بيئة الديمو لا تملك ذاكرة/معالج رسومي كافيًا للنماذج الكبيرة من نوع VLM، مما أدى إلى بطء شديد وتعليق أثناء المعالجة. لذلك تم تثبيت المسار العملي التالي:

1. يحاول النظام استخدام Qari OCR عبر Hugging Face Space.
2. إذا وصلت حصة ZeroGPU أو فشل الاستدعاء، ينتقل تلقائيًا إلى Tesseract حتى يبقى التطبيق قابلًا للاستخدام.
3. في بيئة إنتاجية ذات GPU يمكن ضبط `QARI_OCR_PROVIDER=local` لاحقًا وتنفيذ التحميل المحلي للنموذج المناسب مثل `NAMAA-Space/Qari-OCR-0.4.0-VL-4B-Instruct`.

## نتائج الاختبار

تم اختبار استدعاء Space مباشرة على صورة عربية، ونجح Qari في استخراج النص التالي:

> هذا اختبار للتعرف الضوئي العربي رقيم يستخدم نموذج قاري

بعد ذلك ظهرت رسالة حدود Hugging Face ZeroGPU لأن البيئة لا تحتوي على `HF_TOKEN`. ورغم ذلك نجح اختبار المسار الكامل للباكند والواجهة باستخدام fallback، بما يشمل الرفع، انتظار المعالجة، فتح صفحة المراجعة، حفظ تصحيح يدوي، إرسال التصحيحات، وتنزيل النص المصحح.

## تشغيل الديمو

شغّل الباكند مع الإعدادات التالية:

```bash
cd /home/ubuntu/Raqim/Backend
QARI_OCR_ENABLED=true \
QARI_OCR_PROVIDER=space \
QARI_OCR_FALLBACK_TO_TESSERACT=true \
python3 app.py
```

ثم شغّل الواجهة مع ربطها بعنوان الباكند:

```bash
cd /home/ubuntu/Raqim/my-app
REACT_APP_API_BASE_URL='http://127.0.0.1:5000' npm start
```

إذا كان لديك رمز Hugging Face، ضعه في البيئة قبل تشغيل الباكند:

```bash
export HF_TOKEN='hf_xxx'
```

## تحقق مؤشر المحرك في الواجهة

تمت إضافة مؤشر واضح في صفحة المراجعة يعرض المحرك المستخدم في آخر معالجة. في التحقق الأخير عبر الرابط العام ظهرت البطاقة أعلى صفحة المراجعة بالنص التالي: **المحرك المستخدم: Tesseract fallback**، مع سبب الانتقال: نفاد حصة Hugging Face ZeroGPU عند محاولة تشغيل Qari OCR. هذا يؤكد أن الواجهة أصبحت تعرض حالة المحرك مباشرة للمستخدم بدل الاعتماد على قراءة السجلات.
