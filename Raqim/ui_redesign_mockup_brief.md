# تصور مبدئي لإعادة تصميم واجهة رقيم

هذا المستند يحدد اتجاه تصميم مرئي فقط لإنتاج Mockups مقترحة قبل أي تعديل برمجي. نطاق العمل في هذه المرحلة هو UI/UX فقط، ولا يشمل تعديل مسار OCR أو بنية الباكند أو آليات المعالجة.

## اتجاه الهوية البصرية

سيكون التصميم عربيًا باتجاه RTL بالكامل، بطابع SaaS أكاديمي احترافي. يعتمد التصور على خلفية بيضاء ومساحات واسعة، مع درجات أزرق عميق وهوية هادئة مستوحاة من اسم رقيم بوصفه منتجًا للنصوص والمراجعة اللغوية. سيتم إخفاء كل المعلومات التقنية من واجهة المستخدم النهائي، مثل اسم محرك OCR، اسم النموذج، وقت التشغيل، وأي تفاصيل خاصة بالمطور.

| العنصر | القرار التصميمي المقترح |
|---|---|
| الاتجاه | RTL كامل للواجهات العربية |
| الألوان | أزرق عميق، أزرق سماوي هادئ، أبيض، رمادي بارد، ولمسة ذهبية خفيفة للحالات المميزة |
| الخط | خط عربي حديث مناسب للواجهات مثل IBM Plex Sans Arabic أو Tajawal أو Cairo |
| الأسلوب | واجهة SaaS نظيفة، Cards واضحة، أزرار موحدة، حالات تفاعل بسيطة |
| navbar | اسم/شعار رقيم فقط مع زر دخول أو حالة مستخدم عند الحاجة |
| المعلومات التقنية | مخفية بالكامل عن المستخدم النهائي |

## الشاشات التي سيتم إنتاج Mockups لها

| الشاشة | الهدف من الـ Mockup | ملاحظات المحتوى |
|---|---|---|
| صفحة الدخول والتسجيل | إظهار Login / Signup بالإيميل بتصميم حديث | Mock UI فقط، دون اشتراط قاعدة بيانات الآن |
| صفحة اختيار نوع التصحيح | توضيح الفرق بين التصحيح اليدوي والتصحيح الذكي | بطاقتان واضحتان ومقارنة مبسطة |
| صفحة المراجعة | إعادة تصميم تجربة مراجعة النص بالكامل | Side Panel احترافي بدل popup عند اختيار كلمة |
| Design System Board | توحيد الألوان والأزرار والبطاقات والحالات | لوحة مرجعية مصغرة للعناصر الأساسية |

## تفاصيل تجربة صفحة المراجعة

عند الضغط على أي كلمة، لا يظهر popup. بدلًا من ذلك تظهر لوحة جانبية ثابتة أو Modal واسع من جهة اليمين في RTL. تحتوي اللوحة على الكلمة الحالية، اقتراحات ALLaM، اقتراحات CorpusFilter، حقل تصحيح يدوي، وزر واضح لاعتماد التصحيح. سيتم عرض الاقتراحات كشرائح أو بطاقات قابلة للاختيار، مع تمييز التصحيح الذكي عن التصحيح المعتمد من المدونة/الفلتر.

## حدود هذه المرحلة

لن يتم تعديل الكود الآن. سيتم إنتاج صور Mockup فقط للمراجعة والموافقة. بعد موافقة المستخدم على الاتجاه، يمكن الانتقال إلى تنفيذ التصميم داخل الواجهة الحالية مع الحفاظ على مسار OCR وبنية الباكند كما هي.

## تحديث واجهة التصحيحات بعد ملاحظة المستخدم

اعتمد المستخدم الاتجاه العام للتصاميم، وطلب تعديلًا مهمًا في صفحة التصحيحات. يجب أن تعرض صفحة المراجعة الملف الأصلي والنص المستخرج جنبًا إلى جنب بحيث يستطيع المستخدم مقارنة الكلمة المستخرجة بالكلمة الأصلية في الملف. عند الضغط على كلمة خاطئة داخل النص المستخرج، تظهر اقتراحات التصحيح بطريقة قريبة من Microsoft Word، أي قائمة صغيرة وسريعة بجوار الكلمة أو بطاقة اقتراحات مرتبطة مباشرة بالكلمة المحددة، مع إمكانية فتح لوحة تفصيلية عند الحاجة.

| المتطلب الجديد | القرار التصميمي المقترح |
|---|---|
| عرض الملف الأصلي | لوحة معاينة على أحد جانبي الشاشة تحتوي على صورة/صفحة الملف الأصلي مع تكبير وتصغير وتنقل بين الصفحات |
| عرض النص المستخرج | لوحة مقابلة تعرض النص القابل للمراجعة والتحرير مع تمييز الكلمات المشكوك فيها |
| المقارنة | محاذاة بصرية بين الصفحة الأصلية والنص المستخرج لتسهيل المقارنة اليدوية |
| تفاعل الكلمة الخاطئة | عند الضغط على الكلمة تظهر قائمة اقتراحات صغيرة بجوارها شبيهة بتجربة Word |
| الاقتراحات | تعرض اقتراحات ALLaM و CorpusFilter ضمن قائمة مختصرة، مع خيار "تصحيح يدوي" و"اعتماد" |
| المعلومات التقنية | تبقى مخفية بالكامل عن المستخدم النهائي |

التصور الأنسب هو تقسيم شاشة المراجعة إلى عمودين رئيسيين: **الملف الأصلي** و**النص المستخرج**. تكون اقتراحات Word-like سريعة وخفيفة بجوار الكلمة، بينما يمكن توفير لوحة جانبية اختيارية للتفاصيل المتقدمة عند اختيار "عرض التفاصيل". في التنفيذ الأولي يمكن الاكتفاء بالقائمة السريعة لتقليل التعقيد وتحسين سرعة المراجعة.
